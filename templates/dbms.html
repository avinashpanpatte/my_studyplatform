Chapter 1-2: Introduction & ER Model

1.1 Introduction to DBMS

A Database Management System (DBMS) is software that enables users to define, create, maintain, and control access to databases. It provides an interface between the database and end users or application programs.

1.2 Data Models

  • Hierarchical Model: Tree-like structure with parent-child relationships
  • Network Model: Allows many-to-many relationships
  • Relational Model: Data organized in tables with relationships
  • Object-Oriented Model: Objects and classes with inheritance
  • Object-Relational Model: Combines relational and object-oriented features

1.3 Database Schema and Instance

Schema: Logical structure of the database (metadata)

  • Physical schema: Storage details
  • Logical schema: Tables, views, constraints
  • External schema: User views

Instance: Actual data stored at a particular time

1.4 Entity-Relationship Model

The ER model is a conceptual data model that describes the structure of a database using entities, attributes, and relationships.

1.5 Entities and Attributes

Entity: Real-world object distinguishable from other objects

Entity Set: Collection of similar entities

Attributes: Properties that describe entities

  • Simple vs Composite attributes
  • Single-valued vs Multi-valued attributes
  • Derived attributes
  • Key attributes (Primary key, Candidate key, Foreign key)

1.6 Relationships

Relationship: Association between entities

Relationship Set: Collection of similar relationships

Degree of Relationship:

  • Unary (Recursive): Entity relates to itself
  • Binary: Two entities
  • Ternary: Three entities

1.7 Cardinality Constraints

  • One-to-One (1:1): Each entity in A associates with at most one entity in B
  • One-to-Many (1:N): Each entity in A associates with many entities in B
  • Many-to-Many (M:N): Entities in A associate with many entities in B and vice versa

1.8 Participation Constraints

  • Total Participation: Every entity must participate in the relationship (double line in ER diagram)
  • Partial Participation: Some entities may not participate in the relationship

1.9 Weak Entities

Entities that cannot be uniquely identified by their own attributes and depend on another entity for identification.

Identifying Relationship: Relationship that connects weak entity to its owner entity

1.10 Specialization and Generalization

Generalization: Bottom-up approach, combines similar entities into a higher-level entity

Specialization: Top-down approach, creates subclasses from a higher-level entity

Key Concepts: Data models, schemas, instances, ER diagrams, entities, attributes, relationships

Cardinality: 1:1, 1:N, M:N relationships with examples

Practice Exercises:

  • Draw ER diagram for library management system
  • Convert ER to relational schema
  • Identify weak entities and generalization

Chapter 3-4: Relational Model & SQL

3.1 Relational Model

The relational model represents data as relations (tables). Each relation consists of tuples (rows) and attributes (columns).

3.2 Relational Database Structure

  • Relation: Table with rows and columns
  • Tuple: Row in a relation
  • Attribute: Column in a relation
  • Domain: Set of allowed values for an attribute
  • Degree: Number of attributes in a relation
  • Cardinality: Number of tuples in a relation

3.3 Keys in Relational Model

  • Candidate Key: Minimal set of attributes that uniquely identify a tuple
  • Primary Key: Chosen candidate key to uniquely identify tuples
  • Super Key: Set of attributes that uniquely identify a tuple (may not be minimal)
  • Foreign Key: Attribute that references primary key of another relation
  • Alternate Key: Candidate keys not chosen as primary key
  • Composite Key: Primary key consisting of multiple attributes

3.4 Integrity Constraints

  • Domain Constraint: Values must be from the domain
  • Entity Integrity: Primary key cannot be null
  • Referential Integrity: Foreign key must reference existing primary key or be null
  • Key Constraint: No duplicate values in primary key

3.5 Relational Algebra

Relational algebra is a procedural query language that takes relations as input and produces relations as output.

3.6 Fundamental Operations

  • Select (σ): Selects tuples that satisfy a condition. σ_condition(R)
  • Project (π): Selects specific columns. π_attributes(R)
  • Union (∪): Combines tuples from two relations. R ∪ S
  • Intersection (∩): Tuples common to both relations. R ∩ S
  • Difference (-): Tuples in R but not in S. R - S
  • Cartesian Product (×): All possible combinations of tuples. R × S

3.7 Additional Operations

  • Join: Combines related tuples from two relations
    • Inner Join: Only matching tuples
    • Left Join: All tuples from left, matching from right
    • Right Join: All tuples from right, matching from left
    • Full Join: All tuples from both relations
  • Division (÷): For queries like "find students who have taken all courses"
  • Rename (ρ): Renames relation or attributes

3.8 Structured Query Language (SQL)

SQL is a standard language for managing relational databases. It includes DDL, DML, DCL, and DQL.

3.9 Data Definition Language (DDL)

  • CREATE: Create database objects (tables, views, indexes)
    • CREATE TABLE Student (
    roll_no INT PRIMARY KEY,
    name VARCHAR(50),
    age INT
);
  • ALTER: Modify structure of existing objects
    • ALTER TABLE Student ADD email VARCHAR(100);
  • DROP: Delete database objects
    • DROP TABLE Student;
  • TRUNCATE: Remove all data from table but keep structure

3.10 Data Manipulation Language (DML)

  • INSERT: Add new records
    • INSERT INTO Student VALUES (1, 'John', 20);
  • UPDATE: Modify existing records
    • UPDATE Student SET age = 21 WHERE roll_no = 1;
  • DELETE: Remove records
    • DELETE FROM Student WHERE age < 18;

3.11 Data Query Language (DQL)

SELECT: Retrieve data from database

SELECT column1, column2 FROM table_name
WHERE condition
ORDER BY column_name
GROUP BY column_name
HAVING condition;

3.12 Advanced SQL Concepts

  • Aggregate Functions: COUNT, SUM, AVG, MIN, MAX
  • GROUP BY: Group rows with same values
  • HAVING: Filter groups based on conditions
  • Subqueries: Nested queries
  • Views: Virtual tables based on SELECT queries
  • Indexes: Improve query performance

3.13 Data Control Language (DCL)

  • GRANT: Give permissions to users
  • REVOKE: Remove permissions from users

Relational Algebra: Select, Project, Join, Union, Intersection, Difference

SQL Commands: DDL (CREATE, ALTER, DROP), DML (INSERT, UPDATE, DELETE), DQL (SELECT)

Chapter 5-6: Normalization

5.1 Introduction to Normalization

Normalization is the process of organizing data in a database to reduce redundancy and improve data integrity. It involves decomposing a relation into smaller relations that satisfy certain normal forms.

5.2 Functional Dependencies

A functional dependency (FD) is a constraint between two sets of attributes in a relation. If A → B, then for each value of A, there is exactly one value of B.

  • Trivial FD: B ⊆ A (e.g., AB → A)
  • Non-trivial FD: B ⊈ A
  • Completely non-trivial FD: A ∩ B = ∅

5.3 Types of Functional Dependencies

  • Partial Dependency: A non-key attribute depends on part of the primary key
  • Transitive Dependency: A non-key attribute depends on another non-key attribute
  • Multivalued Dependency: One attribute determines multiple independent values

5.4 Normal Forms

5.4.1 First Normal Form (1NF)

A relation is in 1NF if:

  • All attributes contain atomic (indivisible) values
  • No repeating groups or arrays
  • Each column contains values of a single type

Example: Student table with multiple phone numbers should be split.

5.4.2 Second Normal Form (2NF)

A relation is in 2NF if:

  • It is in 1NF
  • No partial dependencies (non-key attributes depend on the entire primary key)

Elimination: Remove partial dependencies by creating separate tables.

5.4.3 Third Normal Form (3NF)

A relation is in 3NF if:

  • It is in 2NF
  • No transitive dependencies (non-key attributes don't depend on other non-key attributes)

Example: If A → B and B → C, then A → C is transitive dependency.

5.4.4 Boyce-Codd Normal Form (BCNF)

A relation is in BCNF if:

  • It is in 3NF
  • For every FD A → B, A is a superkey

Difference from 3NF: BCNF is stricter and eliminates all redundancy caused by FDs.

5.4.5 Fourth Normal Form (4NF)

A relation is in 4NF if:

  • It is in BCNF
  • No multivalued dependencies
5.4.6 Fifth Normal Form (5NF)

A relation is in 5NF if it cannot be decomposed into smaller relations without loss of information. Also called Project-Join Normal Form (PJNF).

5.5 Normalization Process

  1. Identify all functional dependencies
  2. Check each normal form violation
  3. Decompose relations to eliminate violations
  4. Ensure decomposition is lossless and dependency preserving

5.6 Lossless Join Decomposition

A decomposition is lossless if the natural join of the decomposed relations gives back the original relation without any extra tuples.

Test: Check if the common attributes form a key in at least one of the decomposed relations.

5.7 Dependency Preservation

A decomposition preserves dependencies if all functional dependencies can be checked without joining the relations.

Algorithm: Check if each FD is preserved in at least one decomposed relation.

5.8 Denormalization

Intentionally introducing redundancy to improve query performance. Used when read operations are more frequent than write operations.

Normal Forms: 1NF, 2NF, 3NF, BCNF, 4NF, 5NF

Functional Dependencies: Partial, transitive, multivalued dependencies

Chapter 7-8: Transaction Management

7.1 Introduction to Transactions

A transaction is a logical unit of work that contains one or more SQL statements. Transactions ensure data consistency and integrity in a multi-user environment.

7.2 ACID Properties

ACID properties ensure reliable transaction processing:

  • Atomicity: All operations in a transaction succeed or none do. "All or nothing" principle.
  • Consistency: Transaction brings database from one consistent state to another.
  • Isolation: Concurrent transactions don't interfere with each other.
  • Durability: Changes made by committed transaction persist even in case of system failure.

7.3 Transaction States

  • Active: Transaction is executing
  • Partially Committed: Last statement executed, before commit
  • Committed: Transaction completed successfully
  • Failed: Transaction failed and cannot proceed
  • Aborted: Transaction rolled back due to failure

7.4 Transaction Operations

  • COMMIT: Make transaction changes permanent
  • ROLLBACK: Undo all changes made by transaction
  • SAVEPOINT: Create a point to which transaction can rollback

7.5 Concurrency Control

Concurrency control ensures correct execution of concurrent transactions. Problems without concurrency control:

  • Lost Update: One transaction's update is overwritten by another
  • Dirty Read: Reading uncommitted data that may be rolled back
  • Non-repeatable Read: Reading the same data gives different values
  • Phantom Read: New records appear in subsequent reads

7.6 Serializability

A schedule is serializable if it is equivalent to some serial execution of the transactions.

  • Conflict Serializability: Based on conflicting operations
  • View Serializability: Based on the view of data seen by transactions

7.7 Concurrency Control Protocols

7.7.1 Lock-Based Protocols
  • Two-Phase Locking (2PL):
    • Growing Phase: Acquire locks, no release
    • Shrinking Phase: Release locks, no acquire
  • Types of Locks:
    • Shared Lock (S): Read access
    • Exclusive Lock (X): Write access
7.7.2 Timestamp-Based Protocols

Each transaction has a timestamp. Operations are ordered by timestamp.

  • Thomas Write Rule: Ignore obsolete writes
7.7.3 Optimistic Protocols

Assume no conflicts, validate at commit time.

  • Read Phase
  • Validation Phase
  • Write Phase

7.8 Deadlock

A deadlock occurs when two or more transactions are waiting for each other to release locks.

7.9 Deadlock Prevention

  • Wait-Die: Older transaction waits, younger dies
  • Wound-Wait: Older transaction wounds (aborts) younger, younger waits

7.10 Deadlock Detection and Recovery

  • Detection: Use wait-for graph, cycle indicates deadlock
  • Recovery: Abort transactions to break cycle, select victim based on cost

7.11 Database Recovery

Recovery ensures database consistency after failures.

7.12 Types of Failures

  • Transaction Failure: Logical errors, deadlock
  • System Failure: Power outage, OS crash
  • Media Failure: Disk crash

7.13 Recovery Techniques

  • Log-Based Recovery: Maintain log of all changes
  • Deferred Update: Write changes to log first, update database later
  • Immediate Update: Update database immediately, log changes
  • Shadow Paging: Maintain shadow page table

7.14 ARIES Algorithm

ARIES (Algorithm for Recovery and Isolation Exploiting Semantics) is a recovery algorithm that combines:

  • Write-ahead logging
  • Repeating history during redo
  • Logging changes during undo

ACID Properties: Atomicity, Consistency, Isolation, Durability

Concurrency Control: 2PL, Timestamp ordering, Optimistic protocols

Recovery: Log-based recovery, ARIES algorithm

Chapter 9-10: Indexing & Query Processing

9.1 Introduction to Indexing

Indexing is a data structure technique that allows for faster data retrieval. An index is a copy of selected columns from a table that can be searched efficiently.

9.2 Types of Indexes

9.2.1 Ordered Indexes
  • Primary Index: Built on primary key, dense or sparse
  • Secondary Index: Built on non-key attributes
  • Clustering Index: Index entries point to blocks, not individual records
9.2.2 B-Tree Index

B-Tree is a balanced search tree that maintains sorted data and allows searches, insertions, and deletions in logarithmic time.

  • All leaves at same level
  • Nodes have minimum and maximum number of keys
  • Efficient for disk-based storage
9.2.3 B+-Tree Index

Variant of B-Tree where all values are stored in leaf nodes, and internal nodes contain only keys for navigation.

  • Leaf nodes are linked for range queries
  • All data in leaves, internal nodes for routing
  • Better for range queries than B-Tree
9.2.4 Hash Index

Uses hash function to map keys to bucket addresses.

  • O(1) average search time
  • Not suitable for range queries
  • Static and dynamic hashing

9.3 Index Selection

Factors to consider:

  • Access type (point query, range query)
  • Access time
  • Insertion time
  • Deletion time
  • Space overhead

9.4 Query Processing

Query processing involves translating a high-level query into low-level operations that can be executed by the database engine.

9.5 Steps in Query Processing

  1. Parsing and translation
  2. Optimization
  3. Evaluation

9.6 Query Optimization

Query optimization finds the most efficient execution plan for a query.

9.7 Cost Estimation

  • Access Cost: Cost of accessing data from disk
  • CPU Cost: Cost of performing operations in memory
  • Communication Cost: Cost of transferring data in distributed systems

9.8 Query Execution Plans

A query execution plan shows the sequence of operations to execute a query.

  • Selection operations
  • Join operations
  • Sort operations
  • Projection operations

9.9 Join Algorithms

  • Nested Loop Join: For each tuple in R, scan S
  • Sort-Merge Join: Sort both relations, merge
  • Hash Join: Hash both relations, join partitions
  • Index Join: Use index on join attribute

9.10 Query Optimization Techniques

  • Equivalent Query Transformations: Convert query to equivalent but more efficient form
  • Selection Pushdown: Apply selections as early as possible
  • Join Ordering: Choose optimal order for multiple joins
  • Use of Indexes: Choose appropriate indexes for query predicates

9.11 Materialized Views

Pre-computed results of queries stored as tables. Used to speed up complex queries at the cost of storage and maintenance overhead.

9.12 Query Processing in Distributed Databases

  • Fragmentation and allocation
  • Distributed query optimization
  • Semijoin and bloomjoin techniques

Indexing: B-Tree, B+-Tree, Hash indexing

Query Optimization: Cost estimation, query plans, join algorithms

Practice Exercises:

  • Compare different indexing techniques
  • Analyze query execution plans
  • Optimize slow queries
