📖 Detailed Chapter Breakdown

Chapter 1-2: Introduction & Processes

1.1 What is an Operating System?

An operating system (OS) is a software that acts as an intermediary between the user and the computer hardware. It provides a convenient environment for executing programs and manages computer resources.

1.2 Functions of Operating System

  • Process Management: Creation, scheduling, termination of processes
  • Memory Management: Allocation and deallocation of memory
  • File Management: File creation, deletion, manipulation
  • I/O Management: Management of input/output devices
  • Security: Protection against unauthorized access
  • Networking: Communication between systems

1.3 Types of Operating Systems

  • Batch OS: Jobs processed in batches without user interaction
  • Time-sharing OS: Multiple users share CPU time (Unix, Windows)
  • Real-time OS: Response within strict time constraints
  • Distributed OS: Multiple interconnected computers
  • Embedded OS: Designed for specific hardware (mobile, IoT)

1.4 Process Concept

A process is a program in execution. It is the unit of work in a system.

1.5 Process vs Program

  • Program: Passive entity (code on disk)
  • Process: Active entity (program in execution)

1.6 Process States

  • New: Process being created
  • Ready: Waiting to be assigned to CPU
  • Running: Instructions being executed
  • Waiting/Blocked: Waiting for some event (I/O completion)
  • Terminated: Process finished execution

1.7 Process Control Block (PCB)

PCB is a data structure that stores information about a process:

  • Process ID
  • Process state
  • Program counter
  • CPU registers
  • Memory management information
  • I/O status information
  • Priority

1.8 Context Switching

Context switching is the process of saving the state of a running process and loading the state of another process. It involves:

  • Saving CPU state (registers, PC)
  • Updating PCB
  • Loading new process state

Overhead: Time spent in context switching reduces CPU utilization.

1.9 Process Operations

  • Process Creation: Parent process creates child process
  • Process Termination: Process ends execution
  • Process Suspension: Process moved to secondary storage
  • Process Resume: Suspended process brought back to memory

1.10 Threads

A thread is a lightweight process. Multiple threads can exist within one process, sharing code, data, and files but having their own stack and registers.

1.11 Types of Threads

  • User Threads: Managed by user-level thread library
  • Kernel Threads: Managed by OS kernel

1.12 Multithreading Models

  • Many-to-One: Multiple user threads to one kernel thread
  • One-to-One: Each user thread maps to kernel thread
  • Many-to-Many: Multiple user threads to multiple kernel threads

1.13 Benefits of Threads

  • Responsiveness
  • Resource sharing
  • Economy
  • Scalability

Process Concepts: Process vs Program, Process states, PCB, Context switching

Process Operations: Create, terminate, suspend, resume

Threads: User threads, kernel threads, multithreading models

Practice Exercises:

  • Draw process state diagrams
  • Calculate context switching overhead
  • Implement thread creation in C

Chapter 3-4: CPU Scheduling

3.1 Introduction to CPU Scheduling

CPU scheduling is the process of selecting a process from the ready queue to execute on the CPU. The goal is to maximize CPU utilization and provide fair allocation of CPU time.

3.2 Scheduling Criteria

  • CPU Utilization: Percentage of time CPU is busy
  • Throughput: Number of processes completed per unit time
  • Turnaround Time: Time from submission to completion
  • Waiting Time: Time spent in ready queue
  • Response Time: Time from submission to first response

3.3 Types of Schedulers

  • Long-term Scheduler: Selects processes from job pool to load into memory
  • Short-term Scheduler: Selects process from ready queue to execute
  • Medium-term Scheduler: Handles process swapping

3.4 Scheduling Algorithms

3.4.1 First-Come-First-Served (FCFS)
  • Non-preemptive algorithm
  • Processes executed in arrival order
  • Simple but may cause convoy effect
  • Waiting time depends on arrival order
3.4.2 Shortest Job First (SJF)
  • Non-preemptive: Selects process with shortest burst time
  • Preemptive version: Shortest Remaining Time First (SRTF)
  • Optimal for minimizing average waiting time
  • Requires knowledge of burst time
3.4.3 Round Robin (RR)
  • Preemptive algorithm with time quantum
  • Each process gets CPU for fixed time slice
  • If process doesn't complete, moved to end of queue
  • Fair allocation, good response time
  • Performance depends on time quantum size
3.4.4 Priority Scheduling
  • Each process has priority number
  • Higher priority processes execute first
  • Can be preemptive or non-preemptive
  • May cause starvation of low-priority processes
  • Solution: Aging (increase priority over time)
3.4.5 Multilevel Queue Scheduling
  • Ready queue divided into multiple queues
  • Each queue has its own scheduling algorithm
  • Processes assigned to queues based on properties
  • Fixed priority between queues
3.4.6 Multilevel Feedback Queue
  • Allows process movement between queues
  • Processes start in highest priority queue
  • If time quantum expires, moved to lower priority queue
  • I/O bound processes stay in high priority queues
  • CPU bound processes move to lower priority queues

3.5 Real-time Scheduling

  • Hard Real-time: Missing deadline is catastrophic
  • Soft Real-time: Missing deadline is undesirable but acceptable
  • Rate Monotonic Scheduling (RMS): Shorter period tasks get higher priority
  • Earliest Deadline First (EDF): Task with earliest deadline gets highest priority

3.6 Scheduling in Linux

Linux uses Completely Fair Scheduler (CFS) which:

  • Maintains red-black tree of runnable processes
  • Assigns virtual runtime to each process
  • Selects process with smallest virtual runtime
  • Provides fair share of CPU time

Algorithms: FCFS, SJF, SRTF, Round Robin, Priority, Multilevel Queue

Performance Metrics: CPU utilization, throughput, turnaround time, waiting time, response time

Practice Exercises:

  • Calculate average waiting time for different algorithms
  • Compare scheduling algorithms with Gantt charts
  • Solve numerical problems with arrival times

Chapter 5-6: Process Synchronization

5.1 Race Condition

A race condition occurs when multiple processes access and manipulate shared data concurrently, and the final outcome depends on the order of execution.

5.2 Critical Section Problem

The critical section is a segment of code where shared resources are accessed. The critical section problem is to design a protocol that processes can use to cooperate.

5.3 Requirements for Critical Section Solution

  • Mutual Exclusion: Only one process can execute in critical section
  • Progress: If no process is in critical section, some waiting process should be allowed to enter
  • Bounded Waiting: There should be a limit on how long a process waits to enter critical section

5.4 Peterson's Solution

A software solution for two processes using shared variables:

  • flag[i]: Indicates if process i wants to enter critical section
  • turn: Indicates whose turn it is to enter

Satisfies all three requirements for critical section problem.

5.5 Hardware Solutions

  • Test-and-Set: Atomic instruction that tests and sets a lock variable
  • Compare-and-Swap: Atomic instruction that compares and swaps values
  • Disadvantages: Busy waiting, priority inversion possible

5.6 Semaphores

A semaphore is an integer variable used for signaling between processes. It can be:

  • Binary Semaphore: Values 0 or 1 (mutex)
  • Counting Semaphore: Can take any non-negative value

5.7 Semaphore Operations

  • wait() (P operation): Decrement semaphore, block if negative
  • signal() (V operation): Increment semaphore, wake up waiting process

5.8 Monitors

A monitor is a high-level synchronization construct that encapsulates:

  • Shared data
  • Procedures to manipulate data
  • Synchronization between procedures

Only one process can execute inside monitor at a time.

5.9 Classic Synchronization Problems

5.9.1 Producer-Consumer Problem

Producer produces items and puts in buffer, consumer consumes from buffer.

  • Use bounded buffer
  • Semaphores: empty, full, mutex
  • empty = n, full = 0, mutex = 1
5.9.2 Readers-Writers Problem

Multiple readers can read simultaneously, but writer needs exclusive access.

  • First Readers-Writers: Readers have priority
  • Second Readers-Writers: Writers have priority
  • Semaphores: readCount, wrt
5.9.3 Dining Philosophers Problem

Five philosophers sitting around table, each needs two chopsticks to eat.

  • Can lead to deadlock
  • Solutions: Resource hierarchy, arbitrator, limited philosophers

5.10 Deadlocks in Synchronization

Improper use of semaphores can lead to deadlocks. For example, acquiring semaphores in wrong order.

Critical Section Problem: Mutual exclusion, progress, bounded waiting

Solutions: Peterson's algorithm, hardware solutions, semaphores, monitors

Classic Problems: Producer-consumer, readers-writers, dining philosophers

Practice Exercises:

  • Implement semaphore solutions
  • Solve synchronization problems
  • Debug race conditions

Chapter 7-8: Memory Management

7.1 Memory Hierarchy

Memory is organized in a hierarchy based on speed, cost, and capacity:

  • Registers: Fastest, smallest, most expensive
  • Cache Memory: Faster than RAM, used to speed up access
  • Main Memory (RAM): Primary memory for program execution
  • Secondary Storage: Disk, SSD for long-term storage

7.2 Memory Management Requirements

  • Relocation: Programs should run anywhere in memory
  • Protection: Processes should not access each other's memory
  • Sharing: Allow controlled sharing of memory
  • Logical Organization: Programs organized in modules
  • Physical Organization: Memory organized for efficiency

7.3 Memory Partitioning

  • Fixed Partitioning: Memory divided into fixed-size partitions
    • Equal-size partitions
    • Unequal-size partitions
    • Internal fragmentation
  • Dynamic Partitioning: Partitions created dynamically
    • First-fit, best-fit, worst-fit allocation
    • External fragmentation
    • Compaction to reduce fragmentation

7.4 Paging

Paging is a memory management scheme that eliminates external fragmentation by dividing physical memory into fixed-size blocks called frames and logical memory into same-size blocks called pages.

  • Page: Fixed-size block of logical memory
  • Frame: Fixed-size block of physical memory
  • Page Table: Maps logical pages to physical frames
  • Page Fault: When requested page is not in memory

7.5 Segmentation

Segmentation divides logical memory into variable-size segments, each representing a logical unit (code, data, stack).

  • Segment Table: Maps segments to physical memory
  • Advantages: Logical division, sharing, protection
  • Disadvantages: External fragmentation

7.6 Virtual Memory

Virtual memory allows execution of processes that may not be completely in memory. It provides:

  • Larger address space than physical memory
  • Protection and sharing
  • Efficient memory utilization

7.7 Demand Paging

Pages are loaded into memory only when needed (on page fault).

  • Valid-Invalid Bit: Indicates if page is in memory
  • Page Fault Handling: Trap to OS, find frame, load page, update table

7.8 Page Replacement Algorithms

When no free frame is available, OS must select a victim page to replace.

7.8.1 FIFO (First-In-First-Out)
  • Replace oldest page
  • Simple but may replace frequently used pages
  • Belady's anomaly: More frames can cause more page faults
7.8.2 Optimal
  • Replace page that will not be used for longest time
  • Lowest page fault rate
  • Not implementable (requires future knowledge)
7.8.3 LRU (Least Recently Used)
  • Replace page that has not been used for longest time
  • Good approximation of optimal
  • Requires hardware support for timestamp
7.8.4 Clock Algorithm
  • Efficient approximation of LRU
  • Uses reference bit with circular list
  • Second chance algorithm

7.9 Thrashing

Thrashing occurs when a process spends more time paging than executing. Causes:

  • Too many processes in memory
  • Small page size
  • Low degree of multiprogramming

Solution: Working set model, page fault frequency

7.10 Memory Protection

  • Base Register: Starting address of process
  • Limit Register: Size of process
  • Address Binding: Compile time, load time, execution time

Memory Hierarchy: Registers, cache, main memory, secondary storage

Techniques: Partitioning, paging, segmentation, virtual memory

Page Replacement: FIFO, LRU, Optimal, Clock algorithm

Practice Exercises:

  • Calculate page fault rates
  • Simulate page replacement algorithms
  • Solve memory allocation problems

Chapter 9-10: File Systems & I/O

9.1 File Concepts

A file is a named collection of related information that is recorded on secondary storage. Files provide a logical view of storage.

9.2 File Attributes

  • Name: Human-readable identifier
  • Type: Indicates file content (text, binary, etc.)
  • Location: Device and location on device
  • Size: Current file size
  • Protection: Access control information
  • Time: Creation, last access, last modification

9.3 File Operations

  • Create: Create a new file
  • Open: Prepare file for access
  • Read: Read data from file
  • Write: Write data to file
  • Seek: Reposition file pointer
  • Close: Close file after access
  • Delete: Remove file from system

9.4 File Types

  • Regular Files: Contain user data
  • Directory Files: Contain file information
  • Special Files: Represent I/O devices
  • Pipes: Inter-process communication
  • Symbolic Links: Pointers to other files

9.5 File Structure

  • Unstructured: Sequence of bytes/words
  • Simple Record Structure: Fixed-length records
  • Complex Structures: Variable-length records, trees

9.6 Access Methods

  • Sequential Access: Read/write in order
  • Direct Access: Read/write any block
  • Indexed Access: Use index for faster access

9.7 Directory Structure

Directories organize files and provide naming and access control.

9.7.1 Single-Level Directory
  • All files in one directory
  • Simple but naming conflicts
9.7.2 Two-Level Directory
  • Master directory with user directories
  • Separate directories per user
9.7.3 Tree-Structured Directory
  • Hierarchical structure
  • Absolute and relative paths
  • Current directory concept
9.7.4 Acyclic Graph Directory
  • Allows shared subdirectories
  • Symbolic links or shared directories

9.8 File System Implementation

9.8.1 File Allocation Methods
  • Contiguous Allocation: Files stored in contiguous blocks
    • Fast sequential access
    • External fragmentation
    • Pre-allocation of space
  • Linked Allocation: Each block points to next
    • No external fragmentation
    • Slow random access
    • File allocation table (FAT)
  • Indexed Allocation: Index block contains pointers
    • Fast random access
    • Overhead of index blocks
    • Supports large files

9.9 Free Space Management

  • Bit Vector: Bit for each block (1=free, 0=allocated)
  • Linked List: List of free blocks
  • Grouping: First free block contains addresses of others
  • Counting: Store count of consecutive free blocks

9.10 I/O Hardware

  • Devices: Block devices (disks), character devices (keyboards)
  • Device Controllers: Interface between device and OS
  • Direct Memory Access (DMA): Transfer data without CPU intervention

9.11 I/O Techniques

  • Programmed I/O: CPU waits for I/O completion
  • Interrupt-Driven I/O: Device interrupts CPU when done
  • DMA: Direct transfer between device and memory

9.12 I/O Buffering

  • Single Buffer: One buffer for I/O
  • Double Buffer: Two buffers for overlapping I/O
  • Circular Buffer: Ring buffer for continuous data

File Concepts: File attributes, operations, types, structure

Directory Structure: Single-level, two-level, tree, acyclic graph

Allocation Methods: Contiguous, linked, indexed allocation

Practice Exercises:

  • Design file system structure
  • Calculate disk access times
  • Implement file operations

Chapter 11-12: Deadlocks

11.1 Introduction to Deadlocks

A deadlock is a situation where a set of processes are blocked because each process is holding a resource and waiting for another resource held by another process.

11.2 Deadlock Characteristics

  • Mutual Exclusion: Resources cannot be shared
  • Hold and Wait: Process holding resource waits for another
  • No Preemption: Resources cannot be forcibly taken
  • Circular Wait: Circular chain of processes waiting for resources

11.3 Resource Allocation Graph

A directed graph representing processes and resources:

  • Process Node: Circle representing a process
  • Resource Node: Rectangle representing a resource
  • Request Edge: Process → Resource (process requesting resource)
  • Assignment Edge: Resource → Process (resource assigned to process)

11.4 Deadlock Prevention

Prevent deadlock by ensuring at least one condition cannot hold.

11.4.1 Attack Mutual Exclusion
  • Use spooling for printers
  • Not practical for all resources
11.4.2 Attack Hold and Wait
  • Process must request all resources at once
  • Process releases all resources before requesting new ones
  • Low resource utilization
11.4.3 Attack No Preemption
  • Preempt resources from waiting processes
  • Process releases resources and restarts
  • Complex to implement
11.4.4 Attack Circular Wait
  • Impose ordering on resource types
  • Process requests resources in increasing order
  • Simple and effective

11.5 Deadlock Avoidance

Avoid deadlock by making decisions based on current resource allocation state.

11.6 Banker's Algorithm

A deadlock avoidance algorithm that:

  • Maintains available, allocation, max, need matrices
  • Grants request if system remains in safe state
  • Safe state: Sequence exists where all processes finish
  • Safe sequence: Order in which processes can complete

11.7 Deadlock Detection

Allow deadlocks to occur, then detect and recover.

11.7.1 Single Instance Resources
  • Use wait-for graph
  • Cycle indicates deadlock
11.7.2 Multiple Instance Resources
  • Use banker's algorithm periodically
  • Check for unsafe state

11.8 Deadlock Recovery

  • Process Termination:
    • Abort all deadlocked processes
    • Abort one process at a time until cycle broken
  • Resource Preemption:
    • Select victim process
    • Preempt resources
    • Rollback process to safe state

11.9 Dining Philosophers Problem (Deadlock)

Five philosophers with chopsticks between them. Each needs two chopsticks to eat.

  • Deadlock Solution: Resource hierarchy, allow only 4 philosophers, asymmetric solution

11.10 Livelock

Processes are not blocked but cannot make progress. Example: Two processes changing direction in corridor.

11.11 Starvation

A process waits indefinitely for a resource. Solution: Aging, priority queues.

Conditions: Mutual exclusion, hold and wait, no preemption, circular wait

Handling: Prevention, avoidance (Banker's algorithm), detection, recovery

Practice Exercises:

  • Detect deadlocks in resource allocation graphs
  • Apply Banker's algorithm
  • Solve deadlock prevention problems
